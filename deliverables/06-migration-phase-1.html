PHASE 1: AUDIT & MAP

WEEK 1–2 • 10 MIN READ

WHAT YOUR TEAM SHOULD EXPECT

Your team will be asked to export data and answer questions about how they use each portal. Allow 2-4 hours per team lead for interviews and data pulls.

WHAT THIS PHASE DOES

Before touching anything, you need to know exactly what exists in every portal. This phase is like taking a complete inventory of every room before you start packing boxes for a move. No data is changed, no records are moved. You're simply documenting what's there.

Skipping this step is the single biggest cause of migration problems. Companies that jump straight to migration discover halfway through that two portals have the same field name storing different data, or that a critical integration was missed. The audit prevents all of this.

PROPERTY MAPPING

Different teams often name the same field differently. "Phone Number" in Portal 1 might be "Phone" in Portal 2 and "Contact Phone" in Portal 3. Before migration, you need to map these so the same data doesn't end up in three separate fields in the target portal.

Create a mapping table for every custom property across all portals. For each property, decide: is this the same data under a different name (merge), unique to one portal (keep), or conflicting?

TYPE CONFLICT WARNING

If two portals have a property with the same internal name but different field types (for example, "Status" as free text in one portal but a dropdown in another), the migration tool cannot merge them automatically. You must resolve these before migration — either rename one property or convert its type.

DUPLICATE ESTIMATION

Before migration, estimate how many contacts exist in multiple portals. This tells you the scale of deduplication you'll need and how much you're currently overpaying.

How to do it: Export the email address list from each portal as a CSV file. Use Excel's VLOOKUP or a simple dedup tool to count how many email addresses appear in more than one portal.

Example: Portal 1 has 10,000 contacts. Portal 2 has 8,000 contacts. After comparing email addresses, 3,200 appear in both portals. That means you have 14,800 unique customers — but you're paying HubSpot for 18,000 contact records. That's 3,200 records you're paying for twice.

INTEGRATION INVENTORY

For every third-party app connected to any portal, document:

  • What app is connected?
  • What data does it sync? (contacts, deals, both?)
  • Which direction? (HubSpot → app, app → HubSpot, or both)
  • Can the app connect to a different HubSpot portal?
  • Will reconnecting require new credentials or re-authentication?

DELIVERABLES FROM THIS PHASE

  • Complete portal inventory table
  • Property mapping table with all conflicts identified
  • Duplicate contact count across portals
  • Integration inventory with reconnection plan
  • Email domain and authentication audit
