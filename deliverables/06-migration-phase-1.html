PHASE 1: AUDIT & MAP

WEEK 3–4 • 10 MIN READ

WHAT YOUR TEAM SHOULD EXPECT

Claude connects to each portal via MCP and runs the full audit automatically. Department heads review the output and confirm accuracy. No manual data pulls needed from your team — allow 1-2 hours per team lead for review meetings only.

WHAT THIS PHASE DOES

Before touching anything, you need to know exactly what exists in every portal. This phase builds a complete, verified map of your entire data landscape — every contact, company, deal, ticket, property, workflow, and integration across all portals.

Skipping this step is the single biggest cause of migration problems. Companies that jump straight to migration discover halfway through that two portals have the same field name storing different data, or that a critical integration was missed. The audit prevents all of this.

MCP FAST-TRACKS THIS ENTIRE PHASE

Traditionally, this phase takes 2-3 weeks of manual work — someone exports CSVs, pastes into spreadsheets, and tries to compare fields by hand. That approach is slow and error-prone. With MCP, Claude connects directly to each portal's database via HubSpot's API, queries every object and property programmatically, and generates the complete audit in hours. No exports. No spreadsheets. No human error in the data collection.

PROPERTY MAPPING — AUTOMATED CROSS-PORTAL ANALYSIS

Different teams often name the same field differently. "Phone Number" in Portal 1 might be "Phone" in Portal 2 and "Contact Phone" in Portal 3. Claude queries the property schema from each portal via MCP, compares internal names, display names, and field types programmatically, and generates the full mapping table.

This is database-level schema comparison — Claude reads the property definitions directly from HubSpot's API, not from exported files. It catches conflicts that a manual review would miss.

TYPE CONFLICT — AUTO-DETECTED BY MCP

If two portals have a property with the same internal name but different field types (for example, "Status" as free text in one portal but a dropdown in another), Claude flags this automatically. The migration tool cannot merge conflicting types — these must be resolved before migration by renaming or converting the property type. Claude identifies every conflict; the partner resolves them.

DUPLICATE DETECTION — DATABASE-LEVEL MATCHING

Claude queries the contact records from each portal via MCP and runs cross-portal matching directly against the live data. This is a proper database-level comparison — matching on email address as the primary key, with secondary matching on company domain, phone number, and name combinations for records where email doesn't match exactly.

WHY NOT EXCEL

Manual CSV exports into spreadsheets introduce errors at every step — character encoding breaks on international names, date formats shift between systems, large datasets hit row limits, and VLOOKUP only matches on a single field. For enterprise datasets (10,000+ contacts per portal), this approach is unreliable. MCP queries the source data directly — no exports, no format conversion, no row limits, no encoding issues.

Example: Portal 1 has 10,000 contacts. Portal 2 has 8,000 contacts. MCP cross-reference finds 3,200 exact email matches across both. That means you have 14,800 unique customers — but you're paying HubSpot for 18,000 contact records. That's 3,200 records you're paying for twice.

DATA INTEGRITY ASSESSMENT

Beyond just counting records, Claude runs a full data health check on each portal — identifying problems that will cause issues during migration if not fixed first:

  • Orphaned contacts — contacts with no company association (breaks company-level reporting after merge)
  • Broken associations — deals linked to deleted contacts, tickets referencing removed companies
  • Empty required fields — records missing critical data (email, company name, deal amount)
  • Inconsistent lifecycle stages — same contact at different stages in different portals
  • Stale records — contacts with no activity in 12+ months (clean up before migration, not after)
  • Workflow conflicts — workflows across portals that would trigger on the same records after merge

INTEGRATION INVENTORY

Claude queries each portal's connected apps via MCP and documents:

  • What app is connected and what data it syncs
  • Sync direction (HubSpot → app, app → HubSpot, or bidirectional)
  • Whether the app supports portal reassignment
  • OAuth/API key requirements for reconnection
  • Data dependencies (does the app fail if HubSpot IDs change?)

DELIVERABLES FROM THIS PHASE

All generated by Claude via MCP, reviewed by the partner, confirmed by department heads:

  • Complete portal inventory (live data, not exported snapshots)
  • Property mapping table with all type conflicts flagged
  • Cross-portal duplicate report with confidence scores
  • Data integrity report (orphans, broken links, stale records)
  • Integration inventory with reconnection requirements
  • Email domain and authentication audit
  • Department head sign-off on each portal's data accuracy
CSV EXPORTS — ONLY FOR BACKUP

CSV exports still have a role — as safety backups stored outside HubSpot in case of emergency. But they are never used as the primary audit tool. The audit runs against live data via MCP. Exports are insurance, not analysis.
