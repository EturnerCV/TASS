PHASE 5: EXECUTE MIGRATION

WEEK 7–8 • 10 MIN READ

WHAT YOUR TEAM SHOULD EXPECT

Migration weekend. Some teams may see brief disruption. All new work goes into the target portal.

MIGRATION ORDER

Source portals are migrated one at a time, never in parallel. This is deliberate. Migrating one portal, verifying it completely, and then moving to the next means you always know exactly where a problem came from if something goes wrong.

The recommended order is to start with the smallest portal (fewest records, simplest setup) and finish with the largest. This lets you learn from the first migration and apply those lessons to the bigger, more complex ones.

MIGRATION FLOW
Portal A (smallest) → Import → Verify → Confirm
↓
Portal B (medium) → Import → Verify → Confirm
↓
Portal C (largest) → Import → Verify → Confirm
↓
All portals migrated → Deduplication → Final verification

Do not begin the next portal until the previous one has been fully verified. Rushing through multiple portals at once creates a tangled mess that is extremely difficult to debug.

PER-PORTAL STEPS

For each source portal, follow the same three-step process: run the migration tool, verify the record counts, and spot-check individual records.

RUN THE MIGRATION TOOL

Use the chosen migration tool (selected and tested during the sandbox phase) to import all records from the source portal into the target portal. The tool will map properties from the source to the target based on the mapping you configured during prep.

During the import, the tool will tag every record with the "Source Portal" and "Migration Date" custom properties that were set up in Phase 4. This creates a permanent audit trail showing where each record came from.

VERIFY RECORD COUNTS

After the import completes, compare the numbers. The count of contacts, companies, deals, and tickets in the target portal should match the count from your final source export (plus any records that already existed in the target portal).

OBJECT SOURCE COUNT TARGET COUNT (AFTER) STATUS
Contacts From final export Should match Verify
Companies From final export Should match Verify
Deals From final export Should match Verify
Tickets From final export Should match Verify

If any count is off by more than a handful of records, stop and investigate before proceeding. Common causes include property mapping errors, required fields that were blank in the source, or records that were filtered out by the migration tool's settings.

SPOT-CHECK INDIVIDUAL RECORDS

Pick 10–15 records of each type at random and open them in both the source portal (read-only) and the target portal. Check that:

  • All property values match (name, email, phone, custom fields)
  • Deal amounts and stages are correct
  • Contact-to-company associations are intact
  • Deal-to-contact associations are intact
  • Notes and activity history came across
  • The "Source Portal" and "Migration Date" tags are correct

DEDUPLICATION

After all portals have been imported, the target portal will contain duplicate records. This is expected and normal. The same customer who exists in Portal A and Portal B now has two contact records in the target. The dedup tool installed in Phase 4 handles this.

EMAIL-MATCH CONTACTS

The primary matching rule for contacts is email address. If two contact records share the same email address, they are the same person. The dedup tool identifies these pairs and lets you choose which record to keep as the primary (usually the one with the most complete data) and which to merge in.

When records merge, all activity history, deals, tickets, and notes from both records are combined onto the surviving record. Nothing is lost — the data is consolidated, not deleted.

DOMAIN-MATCH COMPANIES

For companies, the primary matching rule is website domain. If two company records share the same domain (for example, "acme.com"), they are the same company. The dedup tool identifies these and merges them using the same primary-record logic as contacts.

MERGE RULES

Before running the dedup process, define clear merge rules so the tool knows which values to keep when two records conflict:

  • Most recently updated wins — For fields like phone number or address, keep the most recent value since it is more likely to be current.
  • Most complete wins — For fields where one record has data and the other is blank, always keep the data.
  • Combine multi-value fields — For fields like Business Unit tags, combine the values from both records so the merged record shows all brands the customer interacts with.
  • Keep all activity — Never discard notes, emails, calls, or meetings. Merge all activity onto the surviving record.

ASSIGN BUSINESS UNITS

After deduplication, every contact and company in the target portal needs a Business Unit tag. The "Source Portal" property makes this straightforward — any record that came from the Product A portal gets tagged with the Product A Business Unit. Records that came from multiple portals (identified during dedup) get tagged with all relevant Business Units.

This can be done with a bulk update or a simple workflow: if Source Portal equals "Portal 1 — Product A", set Business Unit to "Product A." Run one workflow per source portal.

RECONNECT INTEGRATIONS

Any third-party tools that were connected to the old portals need to be reconnected to the target portal. Common integrations include:

  • Website forms and tracking code (update the HubSpot tracking code on all websites)
  • Email sync (reconnect individual user email accounts)
  • Calendar sync (reconnect meeting scheduling links)
  • Accounting software (reconnect to the target portal)
  • Chat widgets (update embed codes on websites)
  • Zapier or other automation tools (update API keys to point to target portal)

Test each integration after reconnecting to confirm data flows correctly. A broken integration can mean lost leads or missed customer messages.

INVITE USERS

With data migrated, deduplicated, tagged, and integrations reconnected, it is time to bring the team in. Invite users to the target portal and assign them to the teams created in Phase 4.

Consider a phased rollout rather than inviting everyone at once. Start with a small group of power users from each brand who can test their workflows and flag any issues. Once they confirm everything looks right, invite the rest of the team.

Provide each user with a brief orientation: where to find their deals, how to filter by their brand, and where to go if something looks wrong. A 15-minute video walkthrough is usually enough.

POST-MIGRATION VERIFICATION

Before declaring the migration complete, run through this verification checklist. Every item should pass before moving to the email warm-up phase.

VERIFICATION CHECKLIST
  • Record counts match expected totals (contacts, companies, deals, tickets)
  • Spot-check of 10+ records per object type shows correct data
  • All contact-to-company associations are intact
  • All deal-to-contact associations are intact
  • Duplicate records have been merged
  • Business Unit tags are assigned to all records
  • Email domains are authenticated and verified
  • All active integrations are reconnected and tested
  • Workflows are rebuilt and activated
  • Key reports and dashboards are functional
  • Power users from each brand have confirmed their data looks correct
  • All team members are invited and assigned to correct teams
