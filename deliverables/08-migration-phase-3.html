PHASE 3: SANDBOX TESTING

WEEK 3–4 • 10 MIN READ

WHAT YOUR TEAM SHOULD EXPECT

No production impact. Testing happens in an isolated sandbox environment. Your team leads can verify their data looks correct in the test environment before anything real is touched.

WHY TEST FIRST

If something goes wrong in a sandbox, nothing is affected. Your real customers don't get wrong emails, your deal data doesn't get scrambled, and nobody's workflow breaks. You just delete the sandbox and start fresh.

If something goes wrong in production, you're dealing with broken customer data, missed emails, angry teams, and a very stressful weekend. Always test first.

WHAT IS A SANDBOX

Think of it as a "practice copy" of your HubSpot portal. It copies your portal's configuration (properties, pipelines, workflows) but starts with an empty database. You can break things, test migrations, try new workflows — all without touching real data.

HOW TO SET ONE UP

  1. Go to Settings (gear icon top right)
  2. Click Account Management in the left sidebar
  3. Click Sandboxes
  4. Click Create Sandbox (top right)
  5. Choose Standard Sandbox and name it "Migration Test"
ON PROFESSIONAL TIER?

Sandboxes require Enterprise tier. If you're on Professional, you can create a free developer portal at developers.hubspot.com for basic testing. It won't have Business Units, but it can test data flows and API integrations.

RUNNING THE TEST MIGRATION

  1. Install the HubSpot-to-HubSpot Migrator (by Datawarehouse.io) from the HubSpot App Marketplace
  2. Install it in both the source portal and your sandbox
  3. In the migration tool, set the source and destination portals
  4. Start the migration — let it run to completion
  5. Large datasets may take several hours. This is normal.

TEST BUSINESS UNITS

After data is in the sandbox, test the Business Unit setup:

  1. Create a Business Unit for one product line
  2. Assign migrated contacts to that Business Unit
  3. Create a test list filtered by Business Unit — verify correct contacts appear
  4. Create a test workflow with Business Unit as a trigger filter

TEST EMAIL SENDING

  1. Connect at least one brand domain to the sandbox
  2. Create a test email using that brand's template
  3. Send to internal email addresses (yourself, colleagues)
  4. Verify: arrives in inbox (not spam), from address correct, DKIM passes (check email headers)

WHAT DOESN'T MIGRATE

Document everything that didn't come across. These will need to be manually recreated in Phase 4:

  • Marketing email campaign performance data (open/click history)
  • Historical web traffic analytics
  • Reports and dashboards
  • CTAs and landing page designs
  • Knowledge base articles
  • Chatflows and meeting links
  • Playbooks, snippets, and saved views
  • Brand kits and custom surveys
  • Team structure and permissions
